# The business identifier column is named ticket_number, not ticket_id

A Ticket's human-facing identifier — the **Ticket Number** (`Ticket#YYYYMMDD###`, autogenerated at create) — was stored in a column called `ticket_id`, while every child table's `ticket_id` column holds the surrogate uuid (`tickets.id`), and a separate `ticket_number` column existed but was never written or displayed. We drop the dead `ticket_number` column and rename `tickets.ticket_id` → `tickets.ticket_number`, so that across the whole schema `ticket_number` always means the business identifier and `ticket_id` always means the uuid / foreign key.

## Considered options

- **Rename to match the glossary (chosen)** — eliminates the inversion rather than documenting it. After the migration, the two names each carry exactly one meaning everywhere they appear.
- **Keep `ticket_id`, document only (rejected)** — zero migration and zero code churn, but the column name keeps contradicting the term, and every new reader re-hits the trap. The inversion was confusing enough to flip the terminology mid-discussion while we were resolving it — concrete evidence the footgun is real and recurring.

## Consequences

- Migration: drop the unused `ticket_number` column, then `ALTER TABLE public.tickets RENAME COLUMN ticket_id TO ticket_number`. The rename preserves existing data.
- The unique partial index from migration 013 follows the column automatically but keeps its now-stale name `tickets_ticket_id_unique_idx`; recreate it as `tickets_ticket_number_unique_idx` if the name matters (cosmetic).
- A dozen-plus code sites flip `ticket_id` → `ticket_number`: the `Ticket` interface field, `transformTicket`, the `.select`/`.eq`/`.like("ticket_id")` query strings, search, export, the ticket card, the details modal, and the tickets page. The child-table `ticket_id` columns are unchanged — they correctly name the uuid.
- The index is partial (`WHERE ticket_number IS NOT NULL`), so historical Tickets created before the identifier existed may have no Ticket Number; how to present those is a separate decision.
