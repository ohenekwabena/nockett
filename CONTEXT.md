# Nockett

Nockett is a NOC (Network Operations Center) ticketing system: operators log network-incident Tickets, classify them against reference lists, and manage access by Role.

## Language

**User**:
A person with a Nockett account (a row in the `users` table).
_Avoid_: account, member.

**Role**:
A User's access level — exactly one of `user` or `admin` (the enforced `users.role` enum).
_Avoid_: permission; the `roles` / `user_roles` tables (a separate, unused concept — see Flagged ambiguities).

**Admin**:
The elevated Role; the only Role that may invite Users and delete Tickets or attachments.

**Ticket**:
A logged network-incident record — the core unit of work NOC operators track and resolve.

**Ticket Number**:
A Ticket's human-facing identifier — `Ticket#YYYYMMDD###`, autogenerated when a Ticket is created; the id operators read, quote, and search by.
_Avoid_: "ticket ID" — that names the surrogate `id` (a uuid) and the child-table foreign keys, never the Ticket Number. See Flagged ambiguities.

**Reference entity**:
An operator-managed lookup list used to classify Tickets and Users — Category, Priority, Demarcation, Link, Site, Service Type, Detection Source, Traffic Impact, Assignee, Department. Each is a named record (`{ id, name }` today; the descriptor model allows extra typed fields, implemented on demand).
_Avoid_: lookup, master data. Note **Role** is NOT a Reference entity — it is the access enum, and the `roles` table was removed.

## Relationships

- A **User** has exactly one **Role** (`user` or `admin`).
- A **Ticket** has exactly one **Ticket Number** (mandatory); the surrogate `id` uuid is separate and internal.
- Only an **Admin** may invite a new **User** or delete a **Ticket**.
- A **Ticket** is classified by **Reference entities** (its Category, Priority, Site, Link, …); a **User** has a Department.

## Example dialogue

> **Dev:** "We have a `roles` table — does adding a row there create a new access level?"
> **Domain expert:** "No. Access is just **User** or **Admin**, read off `users.role`. The `roles` table doesn't gate anything — it's leftover from a dynamic-roles idea we never finished."

## Flagged ambiguities

- "role" meant both (a) the enforced `users.role` enum and (b) rows in the `roles` reference table managed by the roles-card. **Resolved:** **Role** is the enum (`user` | `admin`); the `roles` / `user_roles` junction is vestigial and slated for removal (NOC-4 / NOC-5).
- "ticket number" / "ticket ID" named several things. The **Ticket Number** is the human `Ticket#YYYYMMDD###` value (autogenerated at create); the surrogate key is the `id` uuid, which is the FK target for every child table (`ticket_comments`, `ticket_notes`, `ticket_attachments`, `ticket_history`, `ticket_sla_timers`). The schema inverted this: the Ticket Number sat in a column called `ticket_id`, each child `ticket_id` held the uuid, and a separate `ticket_number` column was dead (never written or displayed). **Resolved (ADR-0003):** drop the dead `ticket_number`, then rename `tickets.ticket_id` → `ticket_number`, so `ticket_number` always means the Ticket Number and `ticket_id` always means the uuid/FK. Until that migration lands, the column holding the Ticket Number is still named `ticket_id`.
